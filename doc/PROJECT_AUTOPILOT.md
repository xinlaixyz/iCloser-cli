# Project Autopilot 大项目工程自动驾驶设计

日期：2026-05-13
阶段：S6

## 1. 产品目标

Project Autopilot 面向完全不会使用工程工具的用户。用户只需要输入“分析整个项目”“补齐缺失文档”“帮我把项目跑起来”这类自然语言，系统自动完成项目分析、计划拆解、中文选择确认、执行、测试和报告。

核心原则：

- 用户不用猜命令。
- 用户只做选择和授权。
- Agent 负责分析、执行、验证和记录。
- 任何写文件、跑命令、改代码都必须进入中文确认面板。
- 大项目不做一次性失控修改，必须分阶段、小步、可回滚。

## 2. 用户路径

### 路径 A：只分析项目

用户输入：

```text
分析整个项目
```

系统动作：

1. 扫描项目结构、语言、框架、依赖、脚本。
2. 检查 docs、测试、构建入口、模块集中度。
3. 输出项目画像、问题清单、建议下一步。
4. 不写文件、不执行危险命令。

### 路径 B：补齐缺失文档

用户输入：

```text
补齐所有缺失文档
```

系统动作：

1. 先运行 Autopilot 分析。
2. 生成缺失文档候选。
3. 所有文档默认写入 `docs/`。
4. 展示中文文件写入确认面板。
5. 用户确认后写入，并告知绝对路径。

### 路径 C：生成测试计划

用户输入：

```text
检查测试缺口
```

系统动作：

1. 识别测试框架和测试脚本。
2. 按模块输出测试缺口。
3. 先生成测试计划，不直接批量写测试。
4. 用户确认后再按模块生成测试文件。

### 路径 D：低风险自动修复

用户输入：

```text
检查代码质量并修复低风险问题
```

系统动作：

1. 自动分析代码质量问题。
2. 只选择低风险、小范围、可验证的问题。
3. 生成计划和 diff。
4. 用户授权后执行。
5. 自动 build/lint/test。
6. 失败则自动修复或回滚建议。

## 3. S6 研发拆分

### S6.1 Project Autopilot Analyzer

目标：提供 `ic autopilot` / `ic auto`。

能力：

- 只读分析项目。
- 输出项目画像。
- 识别缺失文档。
- 识别测试缺口。
- 给出下一步动作。
- 支持 `--json`。

验收：

- `npm run build`
- `npm run test -- autopilot`
- `npm run smoke:autopilot`
- `npm run smoke:all`

### S6.2 Auto Documentation Writer

目标：根据 Autopilot 报告自动补齐文档草稿。

能力：

- 生成 `docs/README.md`、`docs/PRD.md`、`docs/ARCHITECTURE.md`、`docs/API.md`、`docs/TESTING.md`。
- 只补缺失文档，不覆盖用户已有文档，除非用户确认。
- 写入前必须进入统一 Choice Panel。
- 写入后记录绝对路径，支持“刚才写到哪里了”。

### S6.3 Auto Test Planner

目标：自动生成测试计划，不直接批量修改业务代码。

能力：

- 识别测试脚本和测试框架。
- 按模块列出缺失测试。
- 给出低风险测试生成顺序。
- 后续接入按模块测试生成。

### S6.4 Safe Code Generation Pipeline

目标：低风险自动修复和代码生成。

能力：

- 小步计划。
- 文件锁。
- diff 预览。
- 用户确认。
- 自动验证。
- 报告和记忆更新。

### S6.5 REPL Natural Language Autopilot

目标：REPL 中自然语言触发 Autopilot。

示例：

- “分析整个项目”
- “补齐缺失文档”
- “检查测试缺口”
- “检查代码质量并修复低风险问题”

## 5. 非目标

S6.1/S6.3 不直接写代码、不自动修复。S6.2/S6.5 只允许在用户显式 `--go` 或交互确认后写入 docs/ 文档，并且必须输出磁盘验证回执。原因是大项目必须先建立可信分析、测试规划和确认流程，避免 Agent 失控修改。
