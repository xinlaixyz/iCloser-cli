# iCloser REPL UI 改进计划

## 当前状态

经过 T1-T4 修复后，REPL 核心交互已稳定：启动确认面板统一、AI 分析已集成、表格渲染正确、CJK 对齐准确。但深度审计发现 **30 个 UX 问题**，按严重程度分为 P0/P1/P2 三级。

---

## P0 — 数据安全与功能阻断 (3 项)

### 1. `/clear` 单键 `c` 无确认即销毁全部会话

**根因**: `rl.on('line')` 第 338 行，`c` 直接触发 `state.conversation = []; state.pendingFiles = []`，无任何确认。  
**影响**: 用户在 choice panel 模式下误输 `c`、或不熟悉快捷键的用户，瞬间丢失全部对话历史和待处理文件，不可逆。  
**方案**: 
- `c` 键不再直接清除，改为显示确认：`"确认清除全部对话历史？[y/N]"`
- 或在 choice panel 激活时禁用单键快捷键（`activeChoicePanel` 检查前置）

### 2. `!N` 历史搜索补全功能完全失效

**根因**: `replCompleter` 第 2445 行 `if (!line.startsWith('/')) return [[], line]` 在所有 `!` 逻辑之前提前返回。第 2513-2520 行的 `!` 补全代码永远无法到达。  
**影响**: `!` 历史命令是用户回溯对话的唯一途径，补全失效显著降低效率。  
**方案**: 将 `startsWith('/')` 守卫改为 `startsWith('/') || startsWith('!')`，或把 `!` 补全逻辑移到守卫之前。

### 3. 会话文件明文存储 API Key

**根因**: `saveSession` 保存近 20 条 conversation 消息到 `.icloser/session.json`。首次安装流程鼓励用户粘贴 Key 到对话中，Key 以明文持久化。  
**影响**: API Key 泄露风险 — 任何能读 `.icloser/` 的人都能获取 Key。  
**方案**: 在 `saveSession` 中对 conversation 内容做 Key 模式匹配脱敏（`sk-ant-...`、`sk-or-...` 等前缀识别 + 掩码）。

## P1 — 数据丢失与状态混乱 (8 项)

### 4. 流式输出中用户输入被静默丢弃

**根因**: 第 317 行 `if (streamState !== 'idle') { rl?.prompt(); return; }` — AI 输出时所有输入直接丢弃，无提示。  
**影响**: 用户可能在 AI 输出时输入一整段文字，全部丢失且无任何反馈，造成挫败感。  
**方案**: 
- 显示短暂提示 `"AI 输出中，输入已缓存"` 
- 将输入缓冲到 `pendingInput` 队列，流式结束后自动处理

### 5. Ctrl+C 中断后部分 AI 响应丢失

**根因**: 中断时 `streamState = 'idle'`，但 `fullResponse` 已在 `handleChat` 作用域内累积的内容不会被保存到会话。  
**影响**: 用户已等待并看到部分输出，中断后内容消失，会话历史不完整。  
**方案**: 中断时将 `fullResponse` 追加到 `state.conversation`，标记 `role: 'assistant'` + 截断标记。

### 6. 退出时不显示正在停止的后台进程

**根因**: `shutdownRepl` 第 362-364 行静默遍历 `startedProcesses` 并 `stopStartedProcess`，无用户通知。  
**影响**: 用户不知道哪些服务正在被杀死，可能误以为服务仍在运行。  
**方案**: 退出前打印 `"正在停止 N 个后台进程: platform(8080), ui(3000)..."` 

### 7. 会话过期静默失败

**根因**: `loadSession` 第 2628 行 24 小时过期检查只返回 `false`，无消息。  
**影响**: 用户昨天使用过，今天打开看到空白会话，不知道发生了什么。  
**方案**: 检测到过期时打印 `"上次会话已超过 24 小时，已自动清除"` 

### 8. API Key 向导完成后无明确下一步指引

**根因**: `promptApiKeyWizard` 第 664 行烟雾测试后只显示 "Key 已保存"，测试失败时建议模糊。  
**影响**: 用户不知道接下来该做什么，首次体验中断。  
**方案**: 测试成功 → `"✓ 连接成功！可以直接输入需求开始使用"`；失败 → `"Key 已保存但连接失败：[原因]。输入 /apikey 重新配置，或输入需求先用 mock 模式体验"`

### 9. 配置 JSON 损坏时静默失败

**根因**: `loadGlobalConfig` JSON.parse 被空 catch 吞掉，用户看到空白配置无警告。  
**影响**: 配置文件损坏导致静默切换到 mock 模式，用户不知道为什么 AI 不工作。  
**方案**: catch 块中打印 `"配置文件损坏，已重置为默认配置"` + 备份损坏文件。

### 10. 流式传输折叠时无"正在折叠"指示

**根因**: 代码块 > 30 行后 `codeBlockFolded = true`，输出突然停止，只在块结束时才显示行数。  
**影响**: 用户看到输出突然停止，以为流式卡住了。  
**方案**: 触发折叠时立即打印 `"│ … (后续内容已折叠，完成后显示)"`  

### 11. 离线模式回退在运行时静默

**根因**: `enableOfflineModeIfMissingKey` 只在启动时通知一次，运行时 API Key 失效不通知。  
**影响**: 用户不知道为什么 AI 突然变成了 mock 模式。  
**方案**: 运行时检测到 Key 失效时打印一条一次性的通知。

## P2 — 体验粗糙与润色 (10 项)

### 12. 欢迎画面信息过载

**根因**: theme.ts `welcomeScreen` 输出 platform、Node 版本、locale 时间 — 对日常使用无意义。  
**方案**: 只保留 provider、model、项目名三行。其余移到 `/doctor`。

### 13. 首次使用指南缺少示例和 `/help` 链接

**根因**: `printFirstRunGuide` 说"直接输入需求"但不给示例；说"粘贴 API Key"但不说明往哪粘贴。  
**方案**: 添加一个具体示例（如 `"帮我给登录模块加手机号验证码登录"`）和 `"输入 /help 查看所有命令"`。

### 14. 等待进度条是假的

**根因**: `startWaitingPhase` 的 `WAIT_BAR` 是按时间循环的动画，不反映实际进度。  
**方案**: 去掉假进度条，改为显示 `"等待 AI 响应中 [N s]"` + 脉冲动画。更诚实。

### 15. 状态行不显示 convPhase

**根因**: `convPhase` 追踪 `idle → task_created → task_running → task_completed` 但从未展示。  
**方案**: 在 `printStatusLine` 中添加 `convPhase` 指示器，如 `"◇ 任务执行中"`。

### 16. 状态行在确认模式下丢失上下文信息

**根因**: `pendingSystemOperation` 时状态行只显示操作提示，令牌使用量被隐藏。  
**方案**: 保留令牌信息行，操作提示放在令牌行下方或 choice panel 内部。

### 17. 错误信息无下一步建议

**根因**: 扫描失败、验证失败、启动失败等错误只有 `"失败: [msg]"` 无建议。  
**方案**: 每个错误添加建议行: `"建议: 检查网络连接后重试 /scan"`。

### 18. 网络重试 2 秒黑屏

**根因**: 第 1071 行 `await new Promise(r => setTimeout(r, 2000))` 无视觉反馈。  
**方案**: 显示 `"网络异常，2s 后重试 (1/2)..."` + 倒计时。

### 19. 命令面板区分大小写

**根因**: `cmdCommandPalette` 第 2560 行 `c.name.includes(query)` 区分大小写。  
**方案**: 改为 `c.name.toLowerCase().includes(query.toLowerCase())`。

### 20. 首次 Ctrl+C 后的"退出待处理"状态不可见

**根因**: Ctrl+C 设置 `pendingExitSince` 但无视觉指示器，用户输入任何内容都会重置。  
**方案**: 在提示符中显示 `"[再次 Ctrl+C 退出] ◇ "` 直到用户输入或再次中断。

### 21. 单字母快捷键无提示

**根因**: `y/n/h/s/d/c/w/q` 快捷键在任何地方都不列出。  
**方案**: 在 `/help` 底部添加 `"快捷键: y=确认 n=拒绝 h=帮助 s=扫描 d=差异 c=清除 w=写入 q=退出"`。

## 9 个"不需改动"的设计决策

以下问题经过分析，**不**建议修改，原因如下：

| # | 看似问题 | 不改的理由 |
|---|---------|-----------|
| 22 | 中英文术语混合 (`未知 Provider`) | 中文终端用户习惯中英混排，强行统一反而生硬 |
| 23 | `process.exit(0)` vs 自然退出 | REPL 在 TTY 模式下必须强制退出，自然退出可能挂起 |
| 24 | Tab 补全不支持自然语言 | 自然语言输入空间太大，补全无意义，可能干扰输入 |
| 25 | 多路径文件确认 | 三个路径对应三种场景（chat后/命令/回退），故意设计 |
| 26 | choice panel 模式仍允许快捷键 | 这是功能而非 bug — 高级用户用快捷键，普通用户用面板 |
| 27 | `/clear` 清除历史但不重置 convPhase | convPhase 追踪任务状态，与对话历史独立 |
| 28 | 会话保存近 20 条可能含密码 | P0-3 已通过 Key 脱敏解决 |
| 29 | 欢迎画面信息密度 | P2-12 已解决，精简到 3 行 |
| 30 | 代码块语言标记无高亮 | 终端无语法高亮库，保持当前 `C.dim` 方案已足够 |
