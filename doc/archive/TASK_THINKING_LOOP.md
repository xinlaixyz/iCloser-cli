# iCloser Task Thinking Loop 三步任务循环

日期：2026-05-14
阶段：S6.13

## 1. 核心定义

iCloser 的任务处理不是一次性问答，而是一个可中断、可审计、可验证的循环：

1. 收集上下文
2. 采取行动
3. 验证结果

这三个步骤会不断循环，直到任务完成、用户打断、触发安全停止，或达到最大自动修复轮次。

## 2. 模型与工具分工

| 角色 | 职责 | 不能做什么 |
| --- | --- | --- |
| 模型 | 理解用户目标、推理风险、决定下一步策略、选择要调用的本地能力 | 不能假装已经读过文件，不能假装命令已经执行 |
| 工具 | 读取文件、搜索代码、编辑文件、运行命令、启动服务、验证结果 | 不能自行扩大任务范围，不能绕过用户确认 |
| 验证器 | 判断行动是否成功，提取失败摘要，把证据送回下一轮 | 不能把失败说成成功，不能跳过应有验证 |
| 用户 | 给目标、关键授权、随时打断或换方法 | 不需要提供命令、路径、测试框架细节 |

## 3. 三步循环

### Step 1：收集上下文

系统自动读取当前工作目录、项目索引、相关文件、错误日志、docs、测试、记忆和用户最新输入。

原则：只读动作自动执行，不把“请提供文件内容”这种问题丢给用户。

### Step 2：采取行动

系统使用本地能力真正动手，包括：编辑文件、写入文档、运行命令、启动项目、搜索、生成测试、创建回滚快照。

原则：写文件、系统命令、高风险动作必须进入中文确认面板，用户只需要选择下一步。

### Step 3：验证结果

系统自动检查文件是否真实存在、命令是否成功、build/lint/test/smoke 是否通过、安全扫描是否报警。

原则：失败时自动带着错误信息回到 Step 1；超过限制时给诊断、回滚或换方法选项。


## 3.1 产品流程图逻辑

用户流程图对应的系统流程如下：

```text
用户输入 Prompt
  ↓
收集上下文 Gather Context
  ↓
执行操作 Take Action
  ↓
验证结果 Verify Results
  ├─ 完成 → 任务结束
  ├─ 需要继续 → 回到收集上下文，进入下一轮
  └─ 用户干预 → 补充信息 / 调整方向 / 中断任务
```

用户干预不是额外旁路，而是任意阶段都能触发的系统事件：

| 用户干预 | 系统处理 |
| --- | --- |
| 补充信息 | 记录用户输入，回到“收集上下文” |
| 调整方向 | 更新任务目标，回到“收集上下文”重新规划 |
| 中断任务 | 停止当前循环，保留现场和诊断信息 |

验证结果分支必须明确：

| 验证结果 | 分支 |
| --- | --- |
| pass | 完成，进入任务结束和报告 |
| fail / warn / unknown，仍可自动处理 | 需要继续，回到收集上下文 |
| 超过最大轮次或高风险 | 暂停，给用户中文选择面板 |

## 3.2 三步循环 × 五大工具矩阵

五大工具不是独立列表，而是嵌入三步循环的能力底座：

| 循环步骤 | 必需工具 | 作用 |
| --- | --- | --- |
| 收集上下文 Gather Context | 文件操作、搜索、网络搜索、代码智能 | 读取项目文件、按文件名/正则找代码、查官方文档或错误信息、查看类型错误/定义/引用 |
| 执行操作 Take Action | 文件操作、搜索、执行命令 | 修改代码、新建/重命名文件、定位要改的位置、运行 npm/git/启动服务等命令 |
| 验证结果 Verify Results | 文件操作、搜索、执行命令、代码智能 | 确认文件真实存在、查错误输出、运行 build/lint/test/smoke、查看类型诊断和引用影响 |

五大工具定义：

| 工具类别 | 能力 | 安全/降级规则 |
| --- | --- | --- |
| 文件操作 | 读文件、改代码、新建文件、重命名 | 写入/覆盖/重命名必须限制在项目根目录内；不可用时停止写入，只给文件清单 |
| 搜索 | 按文件名或正则表达式快速找代码 | 只读搜索自动执行；不可用时降级为项目索引和已有上下文 |
| 执行命令 | 运行 npm、git、测试、启动服务器等 | 命令执行前展示目的、原因、影响和中文选择；不可用时不能假装执行过 |
| 网络搜索 | 查文档、查错误信息、查版本兼容性 | 优先官方/一手来源；网络结果默认不持久化 |
| 代码智能 | 类型错误、跳转定义、找引用、语言服务诊断 | 需要对应语言插件；不可用时降级为搜索、扫描和编译错误分析 |

关键原则：模型只负责决定“下一步该用什么工具”，工具层负责真实执行，验证器负责判断执行结果。任何一步缺工具，都必须显式降级或暂停，不允许让模型编造结果。
## 4. 中断与换方法

用户随时可以输入：

- 停止
- 暂停
- 换个方法试试
- 不要改这个文件
- 只看分析，不执行

系统应暂停当前循环，记录用户输入到记忆事件流，并重新规划下一步。

## 5. 研发约束

所有新任务能力必须遵守：

- 不能只调用模型聊天，必须先收集本地上下文。
- 不能让模型声称“无法访问文件系统”，因为工具层可以访问当前项目。
- 工具执行结果必须回传给模型和验证器。
- 每次行动后必须验证。
- 所有用户输入都进入记忆系统，API Key 等敏感输入必须脱敏。
- 循环最多自动执行 3 轮，超过后给用户中文选择：查看原因、回滚、换方法。
