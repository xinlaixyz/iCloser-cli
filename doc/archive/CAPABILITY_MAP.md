# iCloser 能力地图

日期：2026-05-21

产品定位：**本地工程执行器 + Claude Code 替代品 + 长期记忆 Agent Shell**。

这份文档是 iCloser 的产品级能力总目录。它的目的不是罗列命令，而是把分散能力收束成用户能理解的完整产品。

## 一、主流程

```text
理解需求
  -> 调用项目上下文与长期记忆
  -> 调用工具获取证据
  -> 形成计划或答案
  -> 需要时执行与验证
  -> 总结证据并沉淀记忆
```

## 三、P0 产品契约

每一次 AI 工具任务都应该展示五个阶段：

1. **理解需求**：说明本轮用户意图。
2. **调用工具**：展示工具名、目标、结果摘要。
3. **形成结论**：输出最终答案、计划或代码结果。
4. **验证证据**：展示命令输出、网页内容、diff、测试结果；如果没有验证，必须明确说明。
5. **沉淀记忆**：展示本轮复用或保存的证据、规则、偏好。

只要工具调用成功，最终回答必须使用工具结果。工具已经成功后仍回答“让我看看”，就是产品缺陷。

## 四、Mission Cockpit 与黄金路径面板

Mission Cockpit 是 REPL 的主产品界面，黄金路径面板是它的验收状态。它解决的问题是：用户不应该从一串工具日志里猜 AI 到底做了什么，而应该一眼看到本轮任务的完整链路。

任务开始前必须展示：

| 区域 | 必须回答的问题 | 示例内容 |
|---|---|---|
| 任务 | 用户刚才要什么 | “把 Android App 登录页转成 H5 网页” |
| 类型 | 系统判断这是什么工作 | “代码交付 / H5 页面” |
| 目标 | AI 本轮要交付什么 | “可运行网页、diff、验证方式” |
| AI | 当前 Provider / Model | `deepseek / deepseek-v4-flash` |
| 记忆 | 是否使用长期记忆 | “已注入项目规则和相关历史” |
| 确认点 | 用户什么时候需要介入 | “写文件前进入 /diff /write 确认” |

面板必须展示：

| 区域 | 必须回答的问题 | 示例内容 |
|---|---|---|
| 需求 | 用户刚才要什么 | “访问 https://icloser.asia/ 并说明内容” |
| 采用记忆 | 本轮用了哪些项目规则、偏好、历史经验 | “采用 3 条项目规则；没有相关历史” |
| 工具轨迹 | AI 调用了哪些工具、目标是什么、结果是什么 | `web_fetch` 成功，提取网页标题和正文摘要 |
| 结果 | AI 基于工具证据给出的答案或交付物 | “这是 iCloser 的 Web3 钱包入口页面……” |
| 验证 | 结果是否有证据；没有验证时下一步是什么 | “网页抓取成功；未做浏览器截图” |
| 后续 | 还需要用户或系统做什么 | “可继续读取子页面 / 可生成站点摘要” |

验收红线：

- 工具调用成功后，最终回答必须引用工具结果。
- 追问必须优先复用上一轮工具证据，不能无关跳到本地项目代码。
- 面板可以紧凑，但不能省略“工具结果”和“最终结论”的关系。
- 如果工具失败，必须显示失败原因、已尝试的恢复动作和下一步建议。

## 五、命令分组

命令界面按用户心智分组，不再把所有命令平铺展示。

| 分组 | 目标 | 主要入口 |
|---|---|---|
| `setup` | 安装、初始化、模型服务配置、环境诊断 | `ic setup`、`ic doctor`、`ic provider`、`/apikey` |
| `project` | 理解项目、扫描结构、识别技术栈、启动项目 | `ic init`、`ic scan`、`ic project`、`ic android` |
| `ai` | AI 任务规划、工具编排、长任务执行 | `ic`、`ic t`、`ic task-run`、`ic orchestrate` |
| `tools` | 文件、搜索、网页、命令、Git、文档读取等工具能力 | REPL 自动工具、`ic search`、`ic diff explain` |
| `code` | 代码生成、修复、重构、测试生成、影响面分析 | `ic gen`、`ic code`、`ic impact`、`ic verify` |
| `memory` | 长期记忆、项目规则、偏好、候选规则审批 | `ic mem`、`ic mem edit`、`ic mem used`、`ic mem why` |
| `collab` | 团队协作、PR 草稿、commit 草稿、审计、review 输入 | `ic collab`、`ic pr`、`ic commit-draft` |
| `release` | 发布信任、质量门禁、校验和、warning 预算 | `ic release report`、`npm run release:trust`、`npm run release:checksum` |

命令展示原则：

- 首页只展示每组 1 到 2 个主入口。
- `/help` 先展示分组，再展示具体命令。
- 新用户优先自然语言入口，高级用户再进入完整命令表。
- 所有分组都必须能回到同一个黄金路径面板：需求、工具、结果、验证、记忆。

## 六、真实样板任务

后续所有“能力提升”必须至少回归 3 个真实样板任务：

| 样板任务 | 用户目标 | 必须证明的能力 | 当前验收要求 |
|---|---|---|---|
| 网页访问 | “访问这个网址并告诉我内容；继续追问具体是什么” | `web_fetch`、结果复用、追问不跑偏 | 首轮回答必须基于网页内容；追问必须复用上一轮网页证据 |
| Android 需求转 H5 网页 | 将移动端/Android 产品需求转换为可运行 H5 页面 | 需求理解、页面结构、样式实现、交互状态、浏览器验证 | 必须产出可运行 HTML/CSS/JS 或前端组件，并给出浏览器验收证据 |
| 修复 Web 项目 bug | 在 `D:\temp\Codex\AgentFI` 修复真实 bug | 代码理解、diff、验证、风险说明、报告 | 必须有 diff、测试/启动验证、失败修复记录 |

样板任务产物必须统一落到 `doc/golden-path/` 或对应任务文档，至少包含：输入需求、采用记忆、AI 计划、工具轨迹、代码 diff、验证日志、修复记录、最终报告。

评分规则：

- 每个 P0/P1 任务必须说明影响哪些评分维度。
- 每次验收必须记录“开发前分数、目标分数、实际分数、证据路径”。
- 没有真实样板任务证据，只能算内部能力完成，不能算产品体验完成。
- 分数下降必须说明原因，例如真实项目暴露问题、交互不完整、验证缺失。
