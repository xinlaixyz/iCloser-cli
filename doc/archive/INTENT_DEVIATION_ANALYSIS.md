# 项目意图偏离分析

## 原始需求 vs 实际交付

### T1 — 核心价值（没有它产品不成立）

| 需求 | 原始描述 | 实际交付 | 偏离 |
|------|---------|---------|------|
| 深度项目理解 | "AST 级代码图谱：跨文件的调用关系、数据流追踪、影响面预判" | 跨文件调用图 ✅，TS 数据流追踪 ✅，8语言解析 (5 tree-sitter + 3 regex) | ✅ 85% |
| 精准任务执行 | "理解项目风格和约定，生成符合规范的一致代码" | ✅ 风格指纹 + 上下文感知生成 | ✅ |
| 自动验证-修复 | "智能错误定位（不仅显示错误、还定位根因），自动修复循环 ≤ 3 轮" | ✅ 3 轮修复 + 语言工具诊断 | ✅ |

## 总体偏离度

```
T1 核心价值:  6 项需求，4 项 100%，2 项 60-70%
T2 信任壁垒:  4 项需求，全部 100%
T3 体验效率:  5 项需求，全部 100%

总体: 13/15 项 100%，2 项部分完成
偏离度: ~13%
```

## 架构膨胀偏离

原本设计是"精简 CLI 工具"，最终扩展到：
- 文档管理系统 (18 个子命令) — 超出了原始 CLI 范围
- PM 视角功能 (release/roadmap/risk/estimate) — 超出开发者工具定位
- iOS/VSCode/Xcode 插件 — 超出终端 CLI 定位

**正面偏离**：这些扩展增强了产品价值
**负面偏离**：增加了维护负担，拉长了发布周期

## 建议

1. **T1 数据流**：P1，提升代码理解深度
2. **覆盖率门禁**：P1，补全质量保障闭环
3. **功能收敛**：文档/PM/插件作为独立 package，保持核心 CLI 精简
