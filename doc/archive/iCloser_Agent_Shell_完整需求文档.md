# iCloser Agent Shell 完整需求文档

---

## 优先级总览

所有功能按用户价值和产品壁垒分为三个梯队：

| 梯队 | 定位 | 内容 |
|------|------|------|
| **T1 — 核心价值** | 没有它产品不成立 | 深度项目理解（含语言/框架/DB 识别）、精准任务执行、自动验证-修复循环、测试质量保障、上下文压缩管理、多任务并行引擎 |
| **T2 — 信任壁垒** | 区别于通用 AI 聊天 | 修改推理与影响分析、分层可审计记忆系统（短期/任务/长期）、人类审批闭环、安全执行环境 |
| **T3 — 体验与效率** | 日常使用爽不爽 | 简洁 CLI 交互、中文报告、Skill 技能管理、Agent 系统（创建/管理/运行/通信）、多 Provider 适配 |

---

## 一、产品概述

**产品名称：** iCloser Agent Shell

**产品定位：**
一个命令行 AI 工程助手，能深入理解项目代码、精准执行修改任务、自动验证并修复问题，最终生成可审查的变更与报告。


**任务三步循环机制：**
> iCloser 每个任务都按“收集上下文 → 采取行动 → 验证结果”循环推进。模型负责思考与推理，工具负责真实本地操作，验证器负责检查是否成功。失败不会直接丢给用户，而是带着错误信息进入下一轮；用户可随时打断、给新指示或要求换方法。
**核心价值主张：**
> 给 AI 一个任务，它不只是"写出代码"，而是"理解项目 → 精准修改 → 自动验证 → 修到通过为止 → 交付出可审查的变更"。

**目标用户：**
- 需要在大型/不熟悉代码库中做功能开发的工程师
- 需要 AI 辅助但不信任 AI 直接操作代码的团队
- 中文工作环境的企业开发团队

**与通用 AI 聊天工具的本质区别：**
- 通用 AI：你描述代码，它给建议，你手动改
- iCloser Agent：你下任务，它理解项目、精准修改、验证通过、出报告，你只需审查

## 二、用户核心痛点与对应解决方案

### 痛点 1（最高频）：AI 不理解我的项目

> "我让 AI 加一个登录功能，它写了完全不符合项目风格的代码，引入了三个新依赖，还破坏了现有的 session 管理。"

**根因：** 通用 AI 缺乏对项目结构、技术栈、代码风格、模块边界、依赖关系的系统理解。

**解决方案 → 深度项目索引（T1）**
- 不只是列出文件，而是构建 AST 级代码图谱
- 提取：模块依赖图、接口契约、数据流、代码风格指纹
- 每次任务前，自动识别并注入最相关的上下文文件
- 任务执行后，增量更新索引而非重新扫描

### 痛点 2（最致命）：我无法信任 AI 的修改

> "它改了 12 个文件，我花了一下午逐行审查，比我自己写还慢。"

**根因：** 只有 diff 输出，没有修改推理、没有影响分析、没有风险评估。

**解决方案 → 修改推理链 + 影响分析（T2）**
- 每次修改附带"意图-推理-影响"三段说明
- 自动标记高风险修改（跨模块接口变更、数据模型改动）
- 生成变更影响范围图：改了 A → 可能影响 B、C、D
- 用户可逐条审查/确认/拒绝每个修改决策

### 痛点 3（最耗时）：改了坏，坏了改，无限循环

> "AI 改完编译不过，修复后测试挂了，再修复又引入了新 bug，循环三次后我放弃了。"

**根因：** 修改和验证是分离的，验证失败后没有自动修复闭环。

**解决方案 → 自动验证-修复循环（T1）**
- 修改后立即触发验证流水线（编译 → lint → 单元测试 → 集成测试）
- 验证失败时，AI 自动读取失败信息、定位原因、修复、重新验证
- 设置最大重试次数（默认 3 次），超出后保留现场、生成诊断报告
- 目标是：用户下任务时，只有"通过了"或"确实遇到了需要人类判断的问题"两种结果

### 痛点 4（最恼人）：每次都要从头告诉 AI 项目背景

> "昨天让它改了用户模块，今天让它改权限，它完全不记得昨天的决策和约束。"

**根因：** 任务之间无记忆，上下文不延续。

**解决方案 → 项目记忆系统（T2）**
- 每次任务自动记录：修改决策、架构约束发现、用户反馈
- 新任务启动时，自动加载相关历史记忆
- 支持用户显式写入约束：`ic rule "永远不要直接修改数据库 schema"`
- 记忆随项目迭代自动衰减和更新

### 痛点 4.5（团队协作）：AI 做了很多事，但团队不知道哪些能沉淀

> "Agent 执行了任务，也生成了结论，但哪些只是临时上下文，哪些应该成为团队规则？谁确认过？以后还能不能追溯？"

**根因：** 任务执行、记忆沉淀和人类审核没有形成闭环。短期会话、任务日志、长期知识库混在一起，会导致错误经验被长期复用，或重要经验没有沉淀。

**解决方案 → 零知识用户可用的上下人处理模式 + 可审计记忆闭环（T2）**
- 上层人类只负责表达目标、修正误解、确认少数高风险动作
- 下层 Agent OS 负责任务拆解、执行、信息收集、短期记忆、任务记忆、自动压缩和自动风险判断
- 任务完成后，系统自动归档低风险项目记忆；中高风险或跨项目知识才作为候选等待确认
- 审核不能变成用户负担：用户看到的是推荐选项，不是原始日志和内部术语
- 每条记忆附带来源、时间戳、任务 ID、Agent ID、审核状态和版本
- 历史任务可归档为模板，让团队流程逐步沉淀为可复用能力

### 痛点 5（底线问题）：AI 会不会删了我的生产配置？

> "我敢让它改代码，但不敢让它随便执行命令。"

**根因：** AI 操作的爆炸半径不明确。

**解决方案 → 分级执行环境（T2）**
- 三级而非四级（够用且清晰）：**预览模式 / 执行模式 / 特权模式**
- 预览模式（默认）：只分析、只展示即将做的修改，不写任何文件
- 执行模式：可修改源码文件，不可修改配置/脚本/CI文件，不可执行 shell
- 特权模式：需手动确认，用于需要执行构建/测试等命令的场景
- 内置敏感路径清单（`.env`、`credentials.*`、`production.yml` 等），任何模式不可写

## 三、功能需求（按优先级排列）

### T1-1 深度项目索引引擎

项目的"眼睛"——没有准确的项目理解，一切修改都是盲目的。

**输入：** 项目根目录路径
**输出：** 结构化项目知识图谱

**一阶段：项目身份识别（零配置，全自动）**

在 AST 解析之前，先回答"这是什么项目"：

| 识别维度 | 检测内容 | 检测方式 |
|---------|---------|---------|
| **语言** | Go / Rust / TS / JS / Python / Java / Kotlin / C# / PHP / Ruby / Swift 等 | 入口文件 + 特征文件（`go.mod` / `Cargo.toml` / `tsconfig.json` 等） |
| **框架** | React / Vue / Next.js / Django / Spring Boot / Gin / Actix 等 | 依赖文件特征匹配（`package.json` deps / `go.mod` require 等） |
| **数据库** | PostgreSQL / MySQL / SQLite / MongoDB / Redis / Elasticsearch 等 | ORM 配置 + 连接字符串 + migration 文件模式 |
| **构建系统** | npm / yarn / pnpm / cargo / go build / gradle / maven / pip / poetry | 构建配置文件检测 |
| **测试框架** | Jest / Vitest / pytest / go test / JUnit / Cypress / Playwright | 测试文件模式 + 依赖检测 |
| **运行时** | Node.js / Deno / Bun / JVM / CPython / PyPy / Go Native | 引擎/版本锁定文件检测 |
| **部署形态** | Docker / K8s / Serverless / 单体 / 微服务 | Dockerfile / k8s yaml / serverless.yml 检测 |

**识别结果自动写入项目配置，并用于：**
- 选择正确的 AST 解析器（Tree-sitter 语法匹配）
- 选择验证管线默认值（前端 = lint+build+e2e，后端 = compile+unit+integration）
- 匹配对应的代码风格指纹库
- 为 AI 注入"这是一个 Go + Gin + PostgreSQL 项目"的顶级上下文

**二阶段：代码图谱构建（AST 深度解析）**

**核心能力：**
- **AST 解析**：根据识别出的语言，调用对应 Tree-sitter 解析器，提取函数签名、类层次、接口定义、类型关系
- **依赖分析**：模块间 import 关系、外部包依赖及版本、依赖环路检测
- **架构模式识别**：自动检测 MVC/MVVM/微服务/分层架构等组织模式，识别模块边界
- **代码风格指纹**：提取命名规范、缩进风格、错误处理模式、常用工具库
- **接口契约提取**：API 路由定义、请求/响应结构、数据库 schema（含字段类型、索引、外键）
- **数据流追踪**：从入口到持久层的核心数据路径
- **增量更新**：代码修改后只重新分析受影响的部分，不重扫全项目
- **上下文智能检索**：给定一个任务描述，自动检索并注入最相关的 N 个文件（控制在 AI 上下文窗口内）

**验收标准：**
- 语言/框架/数据库识别准确率 ≥ 98%（特征文件匹配）
- 模块依赖识别准确率 ≥ 95%
- 接口契约提取覆盖率 ≥ 90%
- 上下文检索命中率（注入的文件确实被任务用到）≥ 85%

### T1-2 任务执行引擎

产品的"手"——把自然语言任务变成通过验证的代码修改。

**输入：** 自然语言任务描述 + 项目知识图谱
**输出：** 通过验证的代码变更 + diff

**核心流程：**

```
任务描述
  │
  ▼
意图解析 ── 将自然语言分解为具体的修改目标和约束
  │
  ▼
上下文组装 ── 从项目索引检索相关文件 + 相关历史记忆
  │
  ▼
修改方案生成 ── AI 给出修改计划（改哪些文件、怎么改、影响范围）
  │
  ▼
[用户审查点 1] ── 确认修改方案，可调整/拒绝
  │
  ▼
代码执行 ── 逐文件修改
  │
  ▼
验证-修复循环（见 T1-3）
  │
  ▼
[用户审查点 2] ── 审查最终 diff，确认/拒绝/局部调整
  │
  ▼
提交变更 + 记录记忆
```

**关键设计决策：**
- **两个用户审查点**比一个更高效：方案阶段发现问题成本远低于改完代码再发现
- **修改方案先行**：先看到"计划做什么"比直接看 diff 更容易判断方向对不对
- **局部拒绝/调整**：用户可以说"第 3 个文件的修改我不想要，其他保留"

### T1-3 自动验证-修复循环

产品最核心的效率引擎——让 AI 自己把问题修到通过。

**输入：** 代码变更
**输出：** 验证通过的结果，或附带诊断的失败报告

**验证流水线（可配置）：**
```
代码修改完成
  │
  ▼
编译检查 ── 失败 → AI 读取编译错误 → 修复 → 重新编译（最多 3 轮）
  │ 通过
  ▼
Lint 检查 ── 失败 → AI 读取 lint 警告 → 修复 → 重新 lint
  │ 通过
  ▼
单元测试 ── 失败 → AI 分析失败用例 → 判断是修改引入还是测试需更新 → 修复
  │ 通过
  ▼
集成/接口测试（可选）── 失败 → 同上
  │ 通过
  ▼
✅ 验证通过
```

**关键设计：**
- 每轮失败后，AI 拿到的是**具体错误信息**（编译器输出、测试失败堆栈），而不是笼统的"失败了"
- AI 修复时**只能修改本次任务涉及的文件**，防止"为了修一个 bug 重构了整个模块"
- 3 轮仍未通过 → 停止循环，回滚所有更改，输出诊断报告（哪一步过不去、尝试了什么修复、建议人工介入的方向）
- 用户可配置验证管线：前端项目勾选 `lint + build + e2e`，后端项目勾选 `compile + unit + integration`

### T1-4 测试策略与质量保障

自动验证-修复循环解决的是"改完没坏"，但还有一个上游问题：**AI 写的测试本身质量如何？** 如果 AI 为了通过验证而写空壳测试，整个循环就成了摆设。

**AI 生成测试的强制规则：**

| 规则 | 说明 |
|------|------|
| 新代码必须带测试 | 新增函数/类/模块 → 同步生成对应测试文件，不可跳过 |
| 测试不可回退 | AI 不能为了"让测试通过"而删除或弱化已有断言 |
| 修改即追加 | 修改已有函数 → 必须追加或更新该函数对应的测试用例 |
| 覆盖率基线 | 新增代码行覆盖率 ≥ 80%，分支覆盖率 ≥ 60% |
| 空测试检测 | 仅有 `test("x"){}` 无断言的测试视为失败 |

**测试分层决策（AI 自动判断需要哪些测试）：**

```
修改类型分析
  │
  ├─ 纯函数/工具方法       → 单元测试（必选）
  ├─ 涉及 API 接口变更      → + 集成测试
  ├─ 涉及数据模型变更       → + schema 迁移测试
  ├─ 涉及 UI 组件变更       → + 组件渲染测试 + 快照测试
  ├─ 涉及跨模块调用         → + 契约测试
  └─ 涉及权限/安全逻辑      → + 权限边界测试（非授权访问必须 403）
```

**测试执行策略：**
- 增量测试优先：先跑本次修改相关的测试，通过后再跑全量
- 全量回归：增量测试通过后自动触发全量，防止远端影响
- 测试超时熔断：单测超过 30s/集成测试超过 120s 自动终止并报告
- 不稳定测试标记：同一测试在无代码变更下连续失败 2 次 → 标记为 `flaky`，不阻塞通过但输出警告

**验收标准：**
- 新增代码行覆盖率 ≥ 80%
- AI 生成测试的有效断言率 ≥ 90%（断言数 > 0 且非空壳）
- 不引入新的不稳定测试

### T1-5 上下文压缩管理器

大模型上下文窗口有限且昂贵。不能把整个项目 dump 给 AI，也不能每次传太少导致理解偏差。上下文管理器负责**在有限的 token 预算内，让 AI 获得最大信息密度**。

**输入：** 任务需求 + 项目索引 + 历史记忆
**输出：** 压缩后的高质量上下文包

**核心策略：**

```
原始上下文（可能 50K+ tokens）
  │
  ▼
1. 相关性排序 ── 按任务描述的语义相似度，对项目文件打分排序
  │
  ▼
2. 分层压缩：
  │
  ├─ 高相关文件（≥ 0.8 分）── 保留完整源码
  ├─ 中相关文件（0.5–0.8 分）── 保留函数签名 + 关键逻辑 + 接口定义
  ├─ 低相关文件（0.3–0.5 分）── 仅保留模块摘要（文件名 + 导出符号 + 一行职责描述）
  └─ 无关联文件（< 0.3 分）── 不注入
  │
  ▼
3. Token 预算分配：
  ├─ 系统提示词：~2K tokens（固定）
  ├─ 项目元信息：~1K tokens（语言/框架/DB/架构模式摘要）
  ├─ 任务相关代码：~60-80% 预算（核心上下文）
  ├─ 历史记忆：~1-2K tokens（相关过往决策）
  └─ 预留缓冲：~10% 预算（给 AI 响应 + 工具调用结果留空间）
  │
  ▼
4. 动态调整 ── 如果验证失败进入修复循环，压缩策略自动调整：
    第二轮 → 追加失败的测试/编译输出（高优先级）
    第三轮 → 追加第二轮修复的 diff（快速定位反复出错的位置）
```

**结构化压缩方法：**

| 压缩层级 | 输出格式 | Token 占比 | 适用场景 |
|---------|---------|-----------|---------|
| **完整** | 完整源码 | 100% | 本次任务明确要修改的文件 |
| **骨架** | 函数签名 + 关键逻辑摘要 + 接口定义 | ~30% | 依赖但不需要改的模块 |
| **摘要** | 文件职责一句话 + 导出符号列表 | ~10% | 仅需了解存在的模块 |
| **图谱** | dependency-graph 可视化片段 | ~5% | 全局架构导航 |

**记忆压缩（长期）：**
- 单个任务记忆在 ~500 tokens 以内（仅保留决策 + 关键 diff + 验证结果）
- 10 个任务后触发整合：相似任务合并为模式记忆
- 项目级别摘要自动维护（~2K tokens，随项目迭代更新）
- 全局级别摘要（~5K tokens，跨项目共性提取）

**验收标准：**
- 上下文包始终 ≤ 模型上下文窗口的 80%（预留工具调用空间）
- 上下文命中率（注入的文件被 AI 引用）≥ 85%
- 压缩后信息损失率（AI 仍能正确回答关于被压缩代码的问题）≤ 10%

### T1-6 多任务并行引擎

一个项目通常不止一个任务在排队。任务之间可能有依赖，也可能完全独立。并行引擎负责**安全地最大化任务吞吐量**。

**任务状态模型：**

```
QUEUED → SCHEDULED → RUNNING → VERIFYING → COMPLETED
  │          │           │           │
  │          │           │           └──→ FAILED（可重试）
  │          │           │
  │          │           └──→ BLOCKED（等待依赖/文件锁）
  │          │
  │          └──→ CANCELLED（用户取消）
  │
  └──→ 依赖检查失败 → 保持 QUEUED
```

**并行调度策略：**

```
任务队列
  │
  ├─ 1. 依赖分析：构建任务 DAG（有向无环图）
  │     Task-B 依赖 Task-A 的输出 → B 必须等 A 完成
  │     Task-C 与 Task-A 无交集 → C 可与 A 并行
  │
  ├─ 2. 文件冲突检测：
  │     Task-A 修改 src/auth/* → 锁定 src/auth/ 目录
  │     Task-D 也修改 src/auth/login.ts → D 排队等待 A 释放锁
  │     Task-E 修改 src/payment/* → 与 A 无冲突，可以并行
  │
  ├─ 3. 资源池分配：
  │     ┌─────────┐  ┌─────────┐  ┌─────────┐
  │     │ Slot 1  │  │ Slot 2  │  │ Slot 3  │
  │     │ Task-A  │  │ Task-E  │  │ (空闲)  │
  │     │ RUNNING │  │ RUNNING │  │         │
  │     └─────────┘  └─────────┘  └─────────┘
  │     并行槽位数 = min(N-1, CPU 核心数, API 并发限制)
  │
  └─ 4. 优先级调度：
        用户可标记任务优先级：`--priority high|normal|low`
        高优先级任务抢占下一个空闲槽位
        长时间运行的任务可被抢占暂停（checkpoint → 恢复）
```

**并行安全保证：**
- **文件锁机制**：任何任务开始修改文件前，对该文件加排他锁
- **冲突预检测**：任务入队时即预测冲突，比运行时发现成本低
- **合并策略**：如果两个任务确实需要改同一文件，提示用户选择顺序或合并
- **git 分支隔离**：每个并行任务在独立 git worktree 或 branch 上工作（可选）

**命令示例：**
```bash
ic t "增加手机登录"                    # 单个任务
ic t "增加手机登录" "增加邮箱验证"     # 两个并行任务
ic queue                              # 查看全局任务队列
ic queue --watch                      # 实时监控队列状态
ic task cancel <task-id>              # 取消排队中的任务
ic task pause <task-id>               # 暂停运行中的任务
ic task resume <task-id>              # 恢复暂停的任务
```

**验收标准：**
- 无文件冲突的独立任务 100% 可并行执行
- 文件锁冲突检测准确率 100%（零漏报）
- 并行效率：2 个独立任务并行完成时间 ≤ 单任务时间 × 1.3
- 中断恢复：暂停/恢复后不丢失已完成的修改

### T2-1 修改推理与影响分析

产品的"信用体系"——让用户不只是看到改了什么，更理解为什么这样改。

**每次代码修改附带：**
1. **意图（Intent）**：我要达成什么（对应任务描述的哪个子目标）
2. **推理（Reasoning）**：为什么选择这种实现方式（考虑了哪些替代方案、为什么选了当前方案）
3. **影响（Impact）**：这个修改会影响到哪些其他模块/接口/数据流

**影响分析报告结构：**
```
修改文件：src/auth/login.ts (L42-L78)
意图：增加 OAuth 登录流程
推理：复用现有 session 管理模块，新增 OAuth callback handler，
      保持与现有 email/password 登录相同的 session 签发逻辑。
影响：
  ├─ src/auth/session.ts — 间接影响（调用了现有的 createSession）
  ├─ src/middleware/auth.ts — 不受影响（session 格式未变）
  ├─ src/types/user.ts — 需关注（新增了 OAuthProvider 字段）
  └─ tests/auth/login.test.ts — 需新增 OAuth 测试用例
风险等级：中 — 涉及类型变更，但未修改 session 签发逻辑
```

### T2-2 双层级记忆系统

产品的"长期记忆"——让 AI 不仅记得当前项目，还积累跨项目的工程智慧。两层记忆互相补充：项目记忆保证单项目深度，全局记忆保证经验复用。

**层级一：项目级记忆（Per-Project Memory）**

存储于项目的 `.icloser/memory/` 目录。

| 类型 | 内容 | 生命周期 |
|------|------|---------|
| 架构约束 | "用户模块禁止直接访问数据库，必须通过 repository 层" | 永久（用户显式管理） |
| 决策记录 | "上次选择用 JWT 而非 session 是因为需要跨域支持" | 随相关代码存在 |
| 修改历史 | 每次任务的 diff、推理、验证结果（压缩后 ~500 tokens/条） | 永久存档 |
| 用户反馈 | "上次那个实现方式太复杂了，这次简单点" | 按时间衰减 |
| 上下文快照 | 当前模块结构、依赖关系、项目元信息摘要（~2K tokens） | 每次任务后增量更新 |
| 已知问题 | 项目中的已知 bug、技术债、临时 workaround | 随代码修复更新 |

**层级二：全局记忆（Global Cross-Project Memory）**

存储于 `~/.icloser/global-memory/` 目录，跨所有项目共享。

```
~/.icloser/global-memory/
├── tech-stack/             # 按技术栈组织的经验库
│   ├── go-gin-postgres.md  # Go + Gin + PostgreSQL 组合的最佳实践
│   ├── react-nextjs.md     # React + Next.js 常见模式
│   └── python-django.md    # Django 项目的模式积累
├── patterns/               # 跨项目设计模式积累
│   ├── auth-patterns.md    # 认证/授权常见实现模式
│   ├── api-design.md       # RESTful API 设计惯例
│   └── error-handling.md   # 错误处理模式
├── preferences.md          # 用户偏好（代码风格、技术选择倾向、注释语言偏好）
├── known-pitfalls.md       # 踩过的坑（"上次在 Go 项目里用 ORM 导致性能问题"）
└── skills-history.md       # 各 skill 的使用频率和效果评分
```

**全局记忆的积累机制：**
- 每完成 10 个项目任务 → 触发一次跨项目模式提取
- AI 识别不同项目中的共性决策和重复模式，写入对应 tech-stack 记忆
- 用户偏好自动学习：如果你在三个项目中都拒绝了 AI 引入 ORM，全局记忆标记"用户倾向轻 ORM"
- 手动注入：`ic memory global "在所有项目中优先使用 async/await 而非 Promise.then"`

**新项目启动时的记忆加载流程：**
```
ic init
  │
  ├─ 检测到 Go + Gin + PostgreSQL
  │
  ├─ 自动加载全局记忆：
  │   ├─ go-gin-postgres.md → 注入项目上下文
  │   ├─ auth-patterns.md → 如果项目包含 auth 模块
  │   ├─ preferences.md → 用户全局偏好
  │   └─ known-pitfalls.md → Go 相关历史踩坑记录
  │
  └─ 初始上下文已包含跨项目经验，不需要用户从零口述
```

**命令集：**
```bash
# 项目级记忆
ic memory                    # 查看当前项目记忆摘要
ic memory <task-id>          # 查看某次任务的完整记录
ic rule "<约束>"             # 写入永久架构约束
ic rule --list               # 列出所有约束

# 全局记忆
ic memory global             # 查看全局记忆目录
ic memory global <tech>      # 查看特定技术栈的全局记忆
ic memory global "<内容>"    # 写入全局记忆
ic memory search "<关键词>"  # 跨项目和全局记忆搜索

# 记忆维护
ic memory clean              # 清理过期/冗余记忆
ic memory export             # 导出记忆为可读文件
```

**记忆压缩与维护：**
- 项目任务超过 50 个 → 自动触发记忆整合（相似任务合并，过期约束标记）
- 全局记忆每个技术栈保持在 ~5K tokens 以内（超出则摘要压缩）
- 6 个月未访问的记忆自动标记为"休眠"，加载时提示用户确认是否仍适用
- 矛盾检测：新决策与旧记忆冲突时，提示用户确认是否更新

**验收标准：**
- 新项目启动时，同技术栈的历史经验 100% 自动加载
- 项目记忆查询（`ic memory`）输出摘要 ≤ 1 屏
- 跨项目模式提取准确率 ≥ 70%（提取的模式确实在多个项目中适用）

### T2-3 安全执行环境

产品的"护栏"——确保 AI 不会越界。

**三级执行模式：**

| 模式 | 能力 | 默认 |
|------|------|------|
| **预览（Preview）** | 分析项目、展示修改计划、生成 diff 预览 | ✅ 默认 |
| **执行（Execute）** | 写入源码文件、运行验证命令 | 需切换 |
| **特权（Privileged）** | 修改配置文件、执行 git 操作、运行自定义脚本 | 每次需确认 |

**硬性保护（任何模式均生效）：**
- 敏感文件清单：`.env`、`credentials.*`、`*.pem`、`*secret*`、`production*.yml` — **绝对不可写**
- 危险命令清单：`rm -rf /`、`git push --force`、`DROP TABLE`、`chmod 777` — **执行前必须二次确认**
- 操作日志：所有文件修改和命令执行完整记录到 `.icloser/audit.log`
- git 保护：只做 commit，不做 push（除非用户显式授权）

### T2-4 上下人处理模式与可审计记忆闭环

Agent OS 的核心协作模型是：**上层人类表达目标 + 下层 Agent 自动治理 + 分层记忆闭环 + 可审计反馈**。产品必须让完全不懂技术和 Agent 概念的用户也能顺利使用；系统不能把复杂审核责任推给用户。

**角色分工：**

| 角色 | 职责 | 权限边界 |
|------|------|----------|
| 上层人类 | 表达目标、确认少数高风险动作、修正系统误解 | 不需要理解技术细节，拥有最终否决权 |
| 下层 Agent OS | 任务执行、资料收集、初步判断、记忆记录、自动压缩、自动风险评级 | 可自动处理低风险记忆，不能静默执行高风险全局沉淀 |
| 外部接口 | API / 数据源查询 | 默认只读，写入必须经过工具权限和任务授权 |

**记忆分层：**

| 层级 | 当前落点 | 功能 | 生命周期 |
|------|----------|------|----------|
| 短期记忆 | REPL session / task runtime | 会话上下文、即时执行结果、所有用户输入、Agent 输出 | 会话结束或任务完成后归档/丢弃 |
| 任务记忆 | `.icloser/tasks/<task-id>/` + `.icloser/memory.json` | 项目进度、任务状态、执行日志、验证结果 | 持续更新，任务完成后归档 |
| 长期知识库 | `~/.icloser/global-memory/` | 历史经验、流程模板、策略规则、跨项目模式 | 人类审核后持续累积 |
| 外部知识接口 | Provider / API / 搜索工具 | 实时查询、补充上下文 | 默认不持久化 |

**闭环流程：**

```text
Human Task Input
  → Agent Task Parsing & Decomposition
  → Subtasks Assigned to Agents
  → Short-term Memory Logging
  → Execution & Data Collection
  → Task Memory Update
  → Automatic Review / Compression
    ├─ Low Risk → Auto Archive / Auto Approve Project Memory
    ├─ Medium Risk → Batch Candidate, Ask Later
    └─ High Risk → Simple Human Choice
  → Archival / Template Extraction
```

**产品要求：**
- 所有用户输入必须先进入短期记忆原始日志，包括任务描述、普通消息、命令、确认/拒绝、修正意见、审批动作和新增约束
- 与任务相关的用户输入必须在任务结束时归档到任务记忆
- 用户输入中可复用的偏好、规则、流程经验必须由系统先自动压缩、去重、风险评级
- 低风险项目内偏好可自动归档；中高风险或跨项目内容进入长期知识候选
- 全局长期知识不能静默 approved，必须来自明确用户确认或团队策略配置
- API Key、token、密钥、隐私内容必须脱敏保存，或只记录审计摘要
- 所有 Agent 动作、工具调用、文件修改、验证结果必须可追溯
- 用户不能被迫逐条审核记忆；需要确认时必须用简单选择题呈现，并提供推荐默认项
- 用户修改 Agent 结论时，必须记录修正前后版本
- 任务报告中要展示：系统自动处理了哪些记忆、哪些候选等待确认

### 输入框与系统权限操作

REPL 的核心交互不是“让用户输入命令”，而是“让用户做选择”。用户表达目标后，系统负责把可执行动作整理成类似 PowerShell/Claude Code 的审批面板：先展示命令、原因和影响，再让用户按数字授权。

**适用场景：**
- 文件写入、覆盖、撤销
- 依赖安装、项目启动、测试/构建命令
- Git 提交、回滚、分支操作
- 全局记忆沉淀、跨项目规则
- 高风险命令或权限操作

**确认面板必须包含：**
- 命令类型：例如 `PowerShell command` / `Shell command`
- 具体命令：系统将执行的完整命令
- 命令说明：系统准备做什么
- 审批提示：明确显示 `This command requires approval`
- 当前目录：在哪个项目里做
- 原因：为什么判断要做这个
- 影响：会修改文件、启动进程、安装依赖还是写入记忆
- 选择项：`1 Yes`、`2 Yes, and don't ask again for: ...`、`3 No`

**标准形式：**

```text
PowerShell command

    npm run dev
    启动项目

This command requires approval

目录 D:\temp\iCloser2026
原因 用户要求启动当前项目，系统已识别到 package.json 启动脚本。
影响 会启动本地开发服务。

Do you want to proceed?
> 1. Yes
  2. Yes, and don't ask again for: npm run dev
  3. No

输入数字后回车 · 3 取消 · 系统操作不会要求你输入命令
```

**验收标准：**
- 用户看到中文原因/影响解释，同时看到清晰的系统命令审批结构。
- 用户按数字选择后，系统执行并展示结果。
- 结果必须明确：通过、失败、已取消、已启动、已确认存在。
- 所有用户选择都进入短期记忆和审计日志。

### T3-1 CLI 交互设计

**设计原则：** 5 个字符以内开始干活。主命令从 `icloser-agent` 简化为 `ic`（交互模式的 `i` + 代码的 `c`）。

**命令体系（三层结构 — 简洁优先）：**

```bash
# === 核心工作流（每天用）===

ic init                # 初始化项目（自动识别语言/框架/DB）
ic t "增加手机登录"     # 创建并执行任务（t = task，默认预览模式）
ic t "加登录" --go     # --go 直接执行，跳过预览
ic t "加登录" "加支付"  # 同时创建多个独立任务（自动并行）

# === 任务管理（频率高）===

ic st                  # 当前项目任务状态一览（st = status）
ic st <task-id>        # 某任务详细状态
ic diff <task-id>      # 查看 diff
ic y <task-id>         # y = yes，确认应用修改
ic n <task-id>         # n = no，拒绝回滚修改
ic gate <task-id>      # 门禁检查
ic cancel <task-id>    # 取消任务

# === 报告与历史 ===

ic log                 # 任务历史列表
ic log <task-id>       # 某任务完整报告
ic r                   # r = report，最近一次任务的报告

# === 记忆与规则 ===

ic mem                 # mem = memory，项目记忆摘要
ic mem global          # 全局记忆
ic mem global "<内容>" # 写入全局记忆
ic rule "<约束>"       # 写入架构约束
ic rule ls             # ls = list，列出所有约束

# === 配置 ===

ic config              # 查看当前配置
ic config provider claude  # 切换 AI provider
ic config model opus       # 切换模型
ic scan                # 更新项目索引

# === Agent 与 Skill ===

ic agent "<描述>"      # 创建并运行一个 Agent
ic agents              # 查看活跃的 Agent 列表
ic skill ls            # 列出可用 Skill
ic skill use <name>    # 使用某个 Skill
ic skill add <source>  # 安装新 Skill
```

**别名一览（极简模式）：**

| 完整命令 | 别名 | 含义 |
|---------|------|------|
| `ic task` | `ic t` | task |
| `ic status` | `ic st` | status |
| `ic diff` | `ic d` | diff |
| `ic report` | `ic r` | report |
| `ic log` | `ic l` | log |
| `ic memory` | `ic mem` | memory |
| `ic rule ls` | `ic rls` | rule list |
| `ic gate` | `ic g` | gate |
| `ic accept` | `ic y` | yes |
| `ic reject` | `ic n` | no |

**智能默认行为：**
- `ic t "描述"` → 默认预览模式，展示方案后询问是否执行（安全优先）
- `ic t "描述" --go` → 跳过预览直接执行（熟练用户）
- `ic t` 不带参数 → 进入交互式任务输入模式
- 命令歧义自动消解：`ic d` 在无 `<task-id>` 时展示最近一个任务的 diff
- 自然语言容错：`ic 加个登录` ≈ `ic t "加个登录"`（系统自动补全为 task 命令）

**输出风格（中文，简洁）：**
- 成功 → `[✓]`  失败 → `[✗]`  进行中 → `[·]`
- 每次完成任务后只输出一行摘要 + 报告路径，不刷屏

### T3-4 Skill 技能管理系统

Skill 是可插拔的专业能力模块。每个 Skill 封装了特定领域的知识、提示词模板、工具集和验证逻辑。Skill 系统让 iCloser 从"通用 AI 编程助手"变为"可组合的专业工程团队"。

**Skill 模型：**

```
Skill 结构
├── manifest.json       # 技能元信息
│   ├── name: "security-review"
│   ├── version: "1.2.0"
│   ├── description: "安全审查：注入检测、敏感数据扫描、权限边界验证"
│   ├── triggers: ["审查安全", "安全检查", "security audit"]
│   ├── requires: ["project-index"]    # 依赖的其他 skill
│   └── provider: "claude"             # 推荐的 AI provider
├── system-prompt.md    # 系统提示词（定义该 skill 的角色和专业行为）
├── tools/              # 该 skill 专用工具
│   ├── scan-secrets.sh
│   └── check-owasp.py
├── knowledge/          # 领域知识
│   ├── owasp-top10.md
│   └── common-cve.md
└── examples/           # 使用示例
    └── sample-output.md
```

**Skill 分类：**

| 类别 | 示例 | 来源 |
|------|------|------|
| **内置 Skill** | `project-index`、`code-review`、`security-review`、`test-gen`、`api-doc` | 随 iCloser 发布 |
| **社区 Skill** | `react-best-practices`、`go-performance`、`k8s-deploy`、`db-migration` | Skill 市场/GitHub |
| **项目 Skill** | 团队自定义的 `my-company-auth`、`legacy-migration` | `.icloser/skills/` |
| **AI 生成 Skill** | AI 观察重复任务模式后自动生成的 `auto-format-on-save` | `.icloser/skills/auto/` |

**Skill 生命周期：**
```
ic skill add <source>     # 安装（GitHub URL / 本地路径 / 市场名称）
ic skill ls               # 列出已安装
ic skill info <name>      # 查看详情
ic skill use <name>       # 在下一个任务中激活该 skill
ic skill enable <name>    # 设为常驻（每个任务自动加载）
ic skill disable <name>   # 停用常驻
ic skill update <name>    # 更新到最新版本
ic skill remove <name>    # 卸载
```

**Skill 组合链：**

Skill 可以串联为任务流水线。例如一个"安全地添加新功能"流程：

```
ic t "增加用户导出 CSV 功能" --skills security-review,api-doc,test-gen
```

执行顺序：
```
task → [project-index 常驻] → 代码修改 → security-review 审查 →
     → 修复安全问题 → test-gen 生成测试 → api-doc 更新 API 文档 → gate → 交付
```

**现有 Skill 兼容性：**
- 兼容 Claude Code Skill 格式（直接加载 `.md` skill 文件）
- 兼容自定义脚本 Skill（任何可执行文件 + manifest）
- Skill 沙箱：第三方 Skill 默认无权访问网络和文件系统，需用户在 manifest 中授权

**AI 自动生成 Skill：**
- AI 检测到同一类型任务执行 ≥ 3 次 → 提示用户是否自动生成 Skill
- 自动提取：触发关键词、常用提示词片段、工具调用模式
- 用户审核生成结果后才激活

**验收标准：**
- 内置 Skill ≥ 5 个（覆盖扫描、审查、测试、文档、部署）
- 第三方 Skill 安装成功率 ≥ 95%
- Skill 链执行顺序与预期一致，中间失败不跳过后续关键步骤

### T3-5 Agent 系统

Agent 是 iCloser 的一等公民——每个 Agent 是拥有独立上下文、工具集和执行沙箱的自主工作单元。借鉴 Codex 的 Agent 模型和 Claude Code 的子 Agent 系统，提供完整的创建、管理、运行、通信能力。

**Agent 模型：**

```
Agent 实例
├── id: agent-abc123
├── type: task | review | verify | skill | explore
├── status: idle | running | waiting | paused | done | failed
├── context: 独立上下文窗口（可配置 token 预算）
├── tools: 授权使用的工具列表
├── model: 使用的 AI provider + model
├── parent: 父 Agent ID（可为空）
├── children: 子 Agent ID 列表
├── sandbox: 文件系统隔离级别
├── budget: 最大 token 预算 / 最大执行时间
└── result: 执行结果
```

**Agent 类型：**

| 类型 | 职责 | 典型生命周期 |
|------|------|------------|
| **Task Agent** | 执行代码修改任务 | 创建 → 执行 → 验证 → 完成 |
| **Review Agent** | 审查代码变更 | 创建 → 审查 → 输出审查意见 → 完成 |
| **Verify Agent** | 运行验证管线 | 创建 → 编译/测试/检查 → 输出验证报告 → 完成 |
| **Skill Agent** | 执行特定 Skill | 创建 → 加载 Skill 上下文 → 执行 → 返回结果 |
| **Explore Agent** | 搜索和理解代码 | 创建 → 检索 → 分析 → 输出理解报告 → 完成 |
| **Orchestrator Agent** | 编排多个子 Agent | 创建 → 拆分任务 → 分发 → 汇总结果 → 完成 |

**Agent 操作命令：**

```bash
# 创建与运行
ic agent "<描述>"                          # 创建 Agent 执行任务
ic agent "<描述>" --type review            # 创建特定类型 Agent
ic agent "<描述>" --model claude-opus      # 指定模型
ic agent "<描述>" --background             # 后台运行

# 管理与监控
ic agents                                  # 列出所有 Agent
ic agent <id>                              # 查看 Agent 详情
ic agent <id> --watch                      # 实时查看 Agent 执行日志
ic agent pause <id>                        # 暂停
ic agent resume <id>                       # 恢复
ic agent stop <id>                         # 终止
ic agent retry <id>                        # 重试失败的 Agent

# 通信
ic agent tell <id> "<消息>"                # 向运行中的 Agent 发送消息
ic agent ask <id> "<问题>"                 # 向 Agent 提问并等待回复
```

**Agent 间通信机制：**

```
Agent A (Task)                    Agent B (Review)
     │                                  │
     ├─ 完成任务 ──→ 结果写入共享上下文 ──→ 读取结果
     │                                  │
     ├─ ic agent msg <B-id> "请审查" ──→ 收到消息
     │                                  │
     ├────────── 共享记忆总线 ──────────┤
     │   ┌─────────────────────────┐    │
     │   │   Shared Context Pool   │    │
     │   │  - 任务描述              │    │
     │   │  - 代码 diff            │    │
     │   │  - 验证结果              │    │
     │   │  - 审查意见              │    │
     │   └─────────────────────────┘    │
     │                                  │
     └─ 接收审查意见 ←──────── 审查完成 ──┘
```

**Agent 层级与委托：**

```
Orchestrator Agent（用户创建）
  │
  ├─→ Sub-Agent 1: Explore（搜索项目中所有 auth 相关代码）
  ├─→ Sub-Agent 2: Task（修改 auth 模块）
  ├─→ Sub-Agent 3: Verify（验证修改）
  └─→ Sub-Agent 4: Review（审查变更）
       │
       └─→ 汇总结果返回给 Orchestrator → 最终输出给用户
```

**Agent 资源管理：**

| 限制维度 | 默认值 | 说明 |
|---------|-------|------|
| Token 预算 | 100K/Agent | 超过后自动压缩上下文或提示用户 |
| 执行时间 | 10 分钟/Agent | 超时自动暂停，询问用户 |
| 并发 Agent | 3 个 | 超过排队等待 |
| 文件写权限 | 继承创建者 | Task Agent 可写，Review Agent 只读 |
| 网络访问 | 默认关闭 | 需显式授权 |
| 子 Agent 数量 | 最多 5 个 | 防止过度拆分 |

**与 Claude Code Agent 的对标：**

| 能力 | Claude Code | iCloser Agent |
|------|------------|---------------|
| 子 Agent 创建 | ✅ `Agent` 工具 | ✅ `ic agent` 命令 |
| Agent 隔离级别 | worktree | worktree / branch / process |
| Agent 间通信 | 单向（结果返回） | 双向（消息 + 共享上下文） |
| Agent 类型 | general-purpose / explore / plan | task / review / verify / skill / explore / orchestrator |
| 并行 Agent | ✅ | ✅ + 依赖感知调度 |
| Agent 持久化 | 会话内 | 跨会话（可暂停/恢复） |
| Agent 模板 | 无 | 支持 Agent 模板复用 |

**验收标准：**
- Agent 创建到开始执行延迟 ≤ 2s
- Agent 间消息延迟 ≤ 500ms
- 3 个 Agent 并行执行无资源冲突
- Agent 暂停/恢复不丢失上下文
- Orchestrator Agent 拆分的子任务 80% 不需要人工重新分配

## 六、异常处理

| 场景 | 行为 |
|------|------|
| 编译/测试失败 | 进入验证-修复循环，最多 3 轮，仍失败则回滚并输出诊断 |
| AI 返回格式异常 | 重试 1 次，仍失败则记录原始输出到日志，提示用户 |
| 修改触及敏感文件 | 立即阻断，提示用户该文件受保护 |
| 危险命令拦截 | 列出即将执行的命令，要求用户逐条确认 |
| 磁盘空间不足 | 保留已生成的 diff 和日志，提示清理空间 |
| 任务冲突（同一文件被多个任务修改） | 检测并排队，后创建的任务等待先创建的完成 |

## 八、验收标准体系

验收不是笼统的"功能对不对"，而是按四个维度逐项打分。任何维度不通过，任务标记为 `REJECTED`。

### 8.1 功能验收（业务正确性）

| 检查项 | 标准 | 判定方式 |
|--------|------|---------|
| 需求覆盖 | 任务描述中的每个子目标都有对应代码修改 | AI 自检 + 用户审查 |
| 行为正确 | 新增/修改的功能行为符合任务意图 | 测试用例断言 |
| 边界处理 | 空值、越界、异常输入有明确处理逻辑 | 边界测试覆盖 |
| 回退兼容 | 未破坏已有功能的行为 | 全量回归测试通过 |
| 类型安全 | 编译零错误、零隐式 any/unknown 滥用 | TypeCheck 通过 |

### 8.2 技术验收（代码质量）

| 检查项 | 标准 | 判定方式 |
|--------|------|---------|
| 编译通过 | 零编译错误 | 编译命令 exit code = 0 |
| Lint 通过 | 零新增 lint 警告 | lint 命令 exit code = 0，diff 无新增 warning |
| 测试通过 | 新增 + 相关已有测试全绿 | 测试命令 exit code = 0 |
| 覆盖率达标 | 新增代码行覆盖率 ≥ 80%，分支覆盖率 ≥ 60% | 覆盖率报告 diff |
| 架构合规 | 未违反用户定义的架构约束 | 约束检查器 |
| 无破坏性变更 | 公共接口签名未变（除非任务显式要求） | API 契约 diff |
| 依赖变更 | 未引入新依赖（除非任务显式要求） | package.json/go.mod diff |

### 8.3 安全验收

| 检查项 | 标准 | 判定方式 |
|--------|------|---------|
| 敏感文件 | 敏感文件清单中的文件未被修改 | 文件路径白名单检查 |
| 敏感数据 | 代码中无硬编码密钥/密码/Token | 模式匹配扫描 |
| 注入风险 | SQL/命令/模板注入入口点有参数化防护 | 代码模式分析 |
| 权限边界 | 新增接口的鉴权逻辑完整（如有） | 权限测试覆盖 |
| 日志安全 | 日志中无敏感用户数据输出 | 日志语句审查 |

### 8.4 交付验收

| 检查项 | 标准 | 判定方式 |
|--------|------|---------|
| 报告完整性 | 任务报告包含意图、推理、影响、验证结果、回滚方法 | 报告模板字段检查 |
| 变更可追溯 | diff + 推理链一一对应，可逐文件审查 | diff 文件与 reasoning.md 交叉校验 |
| 回滚可行 | `ic rollback <task-id>` 可执行且恢复到任务前状态 | 回滚测试 |
| 记忆已更新 | 任务记录已写入项目记忆 | memory.json 中有对应条目 |

## 九、上线前检查与交付门禁

代码从"改完验证通过"到"可合并/可发布"之间，必须经过一道门禁检查。MVP 阶段虽不做自动部署，但需要给用户一个明确的"可以交付了"的判断。

### 9.1 门禁流程

```
任务完成 + 验证循环通过
  │
  ▼
门禁检查：ic gate <task-id>
  │
  ├── [门禁 1] 测试全绿 + 覆盖率达标
  ├── [门禁 2] 安全扫描无告警
  ├── [门禁 3] 修改推理链完整
  ├── [门禁 4] 报告内容齐全
  ├── [门禁 5] 回滚方案可执行
  ├── [门禁 6] git 状态清洁（无未暂存修改混入）
  │
  ▼
```
