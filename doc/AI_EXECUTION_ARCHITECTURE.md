# AI 执行架构 — 深度分析与提升方案

## 一、当前架构：AI 驱动的执行循环

```
System                         AI (sole controller)
  │                                │
  ├─ 组装上下文 ──────────────────→│
  │                                ├─ 决定调用 read_file
  │←── 执行工具 / 返回结果 ──────│
  │                                ├─ 决定调用 search_code  
  │←── 执行工具 / 返回结果 ──────│
  │                                ├─ 决定调用 search_code (again)
  │←── 执行工具 / 返回"未找到" ──│
  │                                ├─ 决定输出 JSON 变更
  │←── Gate-1 编译验证 ──────────│
  │                                │
  └─ 写入文件 / 完成任务           │
```

**AI 是唯一的决策者。系统是 AI 的仆人。**

问题：
- AI 调用什么工具、按什么顺序、重试几次 — 全部由 AI 决定
- 系统只做两件事：执行 AI 要求的工具、在最终输出时检查编译
- 中间的每一轮，系统没有判断能力

## 二、五个结构性断层

### 断层 1：无工具调用预验证

当前：AI 说 `search_code("不存在的模式")` → 系统直接执行 → 返回"未找到" → AI 再次尝试 → 系统再次执行。

正确的做法：系统在执行前检查 — "这个工具调用是否合理？是否重复了之前的失败模式？是否在策略允许范围内？"

### 断层 2：上下文在循环中退化

当前：
```
Round 1: task + context (3000 tokens)
Round 2: task + context + tools-R1-results (5000 tokens)
Round 3: task + context + tools-R1-R2-results (8000 tokens)
...
Round 5: task + context + 4 轮工具结果 (15000 tokens) → 原始任务被淹没
```

正确的做法：每轮注入的不是完整历史，而是**结构化摘要** — 读了哪些文件、发现了什么、还缺什么信息。

### 断层 3：Agent 编排与工具执行脱节

当前有两个独立的执行路径：
- `executeTask` (index.ts:3103) — AI 工具调用循环，直接写文件
- `AgentManager.orchestrate` (agent/manager.ts) — 创建子 Agent，但子 Agent 不走工具调用循环

这两个路径互不知晓。Agent 编排的结果不会被工具循环使用，工具循环的结果也不会反馈给 Agent 编排。

### 断层 4：验证是终点闸门，不是增量闸门

当前：所有文件在写入前一次性编译验证。1 个失败 = 全部拒绝。

正确的做法：每生成一个文件就验证一个。生成→编译→修复→通过→下一个。增量验证比批量验证成功率高得多。

### 断层 5：AI 不知道自己的能力边界

当前：system prompt 里写了"你有这些工具可用"。但 AI 不知道每个工具的可靠性 — `read_file` 是确定的，`run_command` 可能失败，`web_search` 可能无结果。

正确的做法：系统在每次工具调用后，不仅返回结果，还返回**可靠性元数据** — 这个结果是确定的、可能不完整的、还是需要验证的。

## 三、目标架构：系统驱动的决策循环

```
System (controller)             AI (reasoning engine)
  │                                │
  ├─ 组装上下文                    │
  ├─ 生成工具执行计划 ───────────→│ 分析任务 + 上下文
  │←── 计划 + 优先级 ────────────│ 返回: 需要哪些工具, 按什么顺序
  │                                │
  ├─ 逐个执行计划中的步骤:          │
  │   Step 1: read_file(x) → OK    │
  │   Step 2: search_code(y) → 空  │
  ├─ 空结果 → 自动切换备选 ────→│ 推荐替代方案
  │   Step 2b: read_file(z) → OK   │
  │                                │
  ├─ 收集到足够信息 → 请求合成 ──→│ 基于结构化摘要输出
  │←── JSON 变更契约 ────────────│
  │                                │
  ├─ Gate: 逐文件编译验证          │
  │   file1 → pass → 写入          │
  │   file2 → fail → 修复 → pass   │
  │                                │
  └─ 完成                          │
```

**核心区别：系统决定"何时做什么"，AI 决定"具体怎么做"。**

## 四、实现方案

### 方案 A：轻量级重构（15h，现有架构上叠加控制层）

在现有 `executeTask` 循环上增加三层控制：

**A1. 工具调用预验证器（`tool-guard.ts`）**
- 在执行每个工具调用前检查：
  - 是否在意图策略允许的工具集合内
  - 是否重复了之前 3 次失败的调用（相同工具+相同参数）
  - 如果是重复失败 → 不执行，直接返回"此调用已尝试多次，请换策略"
- 不阻断，只注入反馈

**A2. 结构化上下文压缩器（修改 `currentTask` 拼接逻辑）**
- 不再把 `toolResults.join('\n')` 全文注入
- 改为结构化摘要：
  ```
  本轮探索结果:
  - read_file(src/auth.ts): 成功，280 行，包含 login/logout/register 函数
  - search_code("handler"): 15 条匹配，分布在 8 个文件中
  - search_code("不存在的模式"): 未找到（第 3 次空结果，建议换关键词）
  仍缺失的信息: 数据库 schema、测试覆盖
  ```
- 摘要由系统生成，不是 AI

**A3. 增量验证器（修改 Gate-1）**
- 把 `enforceCodeQuality(changes, ...)` 从批量改为逐文件：
  ```
  for each file in changes:
    write temp → compile → if fail: AI fix → compile → pass
    if pass: write to actual path
  ```
- 每个文件有独立的验证状态
- 一个文件失败不影响其他文件

### 方案 B：架构升级（40h，系统驱动的执行计划）

重写 `executeTask` 的核心循环：

**B1. 执行计划生成（取代当前的 `while (toolRound < MAX)`）**
- 第一轮：AI 接收任务 + 上下文 → 返回执行计划（不是直接调用工具）
- 执行计划结构：
  ```json
  {
    "plan": [
      {"step": 1, "tool": "read_file", "target": "src/auth.ts", "why": "理解现有认证逻辑"},
      {"step": 2, "tool": "search_code", "target": "middleware", "why": "找到中间件注册点"},
      {"step": 3, "tool": "code_intel", "target": "router", "why": "了解路由结构"}
    ],
    "expectedOutput": "了解项目认证架构后生成新增中间件的代码"
  }
  ```
- 系统按计划逐步执行，每步根据结果调整

**B2. 系统决策点**
- 计划执行中，系统在以下情况触发 AI 重新决策：
  - 工具返回空结果（连续 2 次）
  - 工具返回异常（文件不存在、命令执行失败）
  - 收集到足够信息（达到信息阈值）
  - 上下文接近 Token 预算上限

**B3. Agent 统一执行总线**
- 废弃两套执行路径（executeTask + AgentManager.orchestrate）
- 统一为 `ExecutionBus`：
  - 根据任务复杂度自动选择：简单任务 → 串行工具循环，复杂任务 → 多 Agent 并行
  - Agent 不再是独立生命周期，而是并行执行的工具调用组
  - 所有 Agent 共享同一个工具结果池和上下文压缩器

## 五、建议路径

**立即做（方案 A，15h）**：在现有架构上加控制层。不需要重构核心循环，只在工具调用前加预验证、在上下文拼接处加结构化压缩、在验证处改为逐文件。

**下一步做（方案 B 的 B1+B2，25h）**：如果方案 A 的效果明显（工具调用质量提升、上下文退化缓解），再投入重写执行计划循环。

**暂不做（B3，20h）**：Agent 统一总线影响面太大，收益不确定，等 A+B1+B2 验证后再考虑。

## 六、预期效果

| 指标 | 当前 | 方案 A 后 | 方案 A+B1+B2 后 |
|------|------|-----------|-----------------|
| 工具调用有效率 | ~60% | ~80% | ~90% |
| 上下文信息密度 | 低（历史淹没任务） | 中（结构化摘要） | 高（增量注入） |
| 编译一次通过率 | ~50% | ~70% | ~85% |
| AI 重复失败次数 | 常见 | 系统阻止 | 系统阻止 + 自动切换 |
| 端到端任务成功率 | ~50% | ~65% | ~80% |
