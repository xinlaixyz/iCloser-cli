# 任务驾驶舱 P0-P2 升级执行文档

日期：2026-05-22  
定位：把 REPL 从“工具流水账”升级为“AI 工作指挥系统”  
目标：普通用户不懂代码，也能通过任务驾驶舱看懂 AI 如何规划、取证、交付、验证和恢复。
当前状态：P0-P2 第一轮已落地，测试和构建通过。

## 1. 总目标

任务驾驶舱必须回答 6 个问题：

1. AI 是否理解了我的真实需求？
2. AI 准备用哪些步骤完成？
3. AI 调用了哪些工具，证据来自哪里？
4. AI 最终交付了什么？
5. 结果是否经过证据或命令验证？
6. 如果失败，下一步怎么恢复？

完成后，任务驾驶舱目标体验分：

| 维度 | 目标分 |
| --- | ---: |
| 工具执行可视化 | 9.2 |
| AI 工具使用能力 | 9.0 |
| 追问连续性 | 9.3 |
| 失败恢复体验 | 9.0 |
| 非技术用户可理解性 | 9.5 |
| Claude Code 对标体验 | 9.0+ |

## 2. P0：驾驶舱闭环与失败恢复

### P0-1 合并 Golden Path 与任务结果面板

问题：

- 当前 `任务结果` 和 `Golden Path` 分开展示，信息重复。
- 用户会看到两套阶段状态，增加界面噪音。

目标：

- `renderMissionResult()` 成为唯一任务结束面板。
- 面板内直接展示：理解、取证、交付、验证、记忆。
- REPL 不再额外打印 `renderGoldenPathState()`。

验收：

- 完成任务后只出现一个结果面板。
- 面板仍能展示证据数、工具数、AI 轮数、token、来源、下一步。

### P0-2 失败恢复面板

问题：

- 失败原因只截断 120 字，容易把关键恢复动作截掉。
- 用户不知道应该重试、换模型、缩小任务，还是继续用已有证据。

目标：

- 失败时展示：
  - 失败摘要
  - 关键错误
  - 恢复动作
  - 是否已有可用证据

验收：

- DeepSeek 超时要显示 `切换 deepseek-v4-flash`、`缩小任务范围`、`用已有证据生成兜底版` 等动作。
- 非 git、网络、Provider、JSON 解析错误要有不同恢复建议。

### P0-3 阶段状态真实

问题：

- 当前阶段状态偏“乐观”，有些任务失败后仍显示若干阶段完成。

目标：

- 阶段状态按任务状态计算：
  - 理解：创建任务即完成
  - 取证：有工具或 Evidence 才完成
  - 交付：有 finalResponse 或 patch 才完成
  - 验证：代码/启动看命令，网页/分析看成功证据
  - 记忆：有 memoryApplied 才完成

## 3. P1：结果模板与追问证据复用

### P1-1 专用结果模板

不同任务不能用同一套结果话术。

| 任务类型 | 结果模板 |
| --- | --- |
| 网页访问 | 标题、来源、主要内容、直接回答、可追问点 |
| 投资/市场研究 | 公司概况、市场、融资/估值、竞品、风险、尽调缺口、置信度 |
| 代码修改 | 影响面、补丁文件、风险、验证命令、下一步 |
| 项目启动 | 环境、启动命令、运行状态、失败原因、恢复动作 |
| 发布验收 | 质量门禁、测试、风险、发布物、阻塞项 |

验收：

- 投资报告不再只输出几行泛泛结论。
- 网页访问不再回答“让我访问看看”。
- 代码任务必须明确 diff、验证和下一步。

### P1-2 追问复用上一轮证据

问题：

- 用户问“具体是什么呀”“详细点”时，AI 可能跳到本地代码项目。

目标：

- 追问模式必须优先使用 `lastToolEvidence`。
- 追问模式禁止重新分析当前项目，除非用户明确要求。

验收：

- 上轮访问网页后，追问“具体是什么呀”，回答必须基于上轮网页内容。
- 不允许无故读取本地 `package.json`、`src/cli/index.ts`。

### P1-3 Evidence Store 作为唯一证据源

目标：

- 工具结果先进入 Evidence Store。
- 用户结果面板和 AI 后续追问都从 Evidence Store 取证。
- 对外展示时只显示摘要和来源，避免刷屏。

## 4. P2：完成评分系统

每个任务完成后自动输出一行任务评分，让后续迭代不再散打。

评分维度：

| 维度 | 权重 | 标准 |
| --- | ---: | --- |
| 任务识别 | 15 | 类型是否识别正确 |
| 工具使用 | 15 | 是否使用了合适工具 |
| 证据质量 | 20 | 是否有来源、文件或命令支撑 |
| 合成质量 | 20 | 是否回答用户真正问题 |
| 验证闭环 | 15 | 是否完成证据/命令验证 |
| 恢复路径 | 10 | 失败时是否给出可执行下一步 |
| 记忆使用 | 5 | 是否正确召回项目规则 |

验收：

- 每个任务结果面板显示 `评分 88/100`。
- 分数低于 75 必须给出主要扣分原因。
- 失败任务也能评分，不能只显示失败。

## 7. 已落地改动

### P0 已完成

- REPL 任务结束只保留一个 `任务结果` 面板，不再额外打印重复的 Golden Path 面板。
- 结果面板内统一展示：理解、取证、交付、验证、记忆、证据、来源、模板、评分、下一步。
- 失败信息拆成：
  - `失败`：短摘要
  - `关键`：关键错误
  - `恢复`：可执行恢复动作
- DeepSeek 超时会提示用已有证据生成兜底版，必要时切换 `deepseek-v4-flash`。
- JSON 解析、非 git、Provider/API Key 等失败有不同恢复动作。

### P1 已完成

- 任务分类补强：投资、融资、估值、市场、商业、竞品、尽调类输入稳定归入 `analysis`。
- AgentTaskLoop 注入结果模板要求：
  - 网页访问：标题、来源、主要内容、直接回答、可追问点。
  - 投资/市场研究：公司概况、市场机会、融资/估值线索、竞品、核心风险、尽调缺口、置信度。
  - 项目分析：现状、关键问题、风险等级、优化路径、下一步验收。
  - 发布验收：质量门禁、测试结果、风险、发布物、阻塞项、下一步。
  - 记忆任务：召回规则、冲突点、候选新增规则、是否需要写回。
- 结果面板展示当前任务使用的结果模板。

### P2 已完成

- 新增任务评分系统 `evaluateMissionScore()`。
- 每轮任务结果面板显示：

```text
评分    88/100 · 良好
扣分    缺少命令验证；未使用相关记忆
```

- 评分维度覆盖：
  - 任务识别
  - 工具取证
  - 证据来源
  - 合成质量
  - 验证闭环
  - 失败恢复
  - 记忆使用

## 8. 下一轮建议

P0-P2 已形成可用闭环，但距离 9.5 分还需要继续做三件事：

1. 把评分从“结果面板评估”升级成“任务报告持久化”，写入 `.icloser/agent-tasks/<taskId>/score.json`。
2. 投资报告增加正文结构化补全器，AI 输出缺少“融资/竞品/尽调缺口”时自动提示补齐。
3. 网页访问和投资研究增加来源可信度分级，例如官网、媒体、数据库、搜索摘要分别标注。

## 9. T1-T2 追加进展

### T1：结果质量守门器

已新增 `src/core/result-quality-gate.ts`。

能力：

- 按任务类型定义必填字段。
- 检查最终回答是否缺少模板字段。
- 输出质量分、状态、已覆盖字段、缺失字段和下一步补齐动作。
- 驾驶舱结果面板新增：

```text
质量    58/100 · 不通过
缺口    公司概况、市场机会、竞品分析、尽调缺口
```

### T2：真实任务回归套件

已新增 `src/core/real-task-regression-suite.ts` 和 `tests/real-task-regression-suite.test.ts`。

当前覆盖 4 类客户直面任务：

| 样板任务 | 目标 |
| --- | --- |
| 网页访问：解释 iCloser 官网内容 | 防止“查了网页但不回答” |
| 投资研究：补齐 iCloser 投资报告 | 防止报告缺融资、竞品、尽调缺口 |
| 代码交付：修复 Web 项目 bug | 防止代码任务没有影响面、diff、验证 |
| 项目启动：识别环境并启动 | 防止只解释不启动、不恢复 |

这些样板任务不依赖外部网络和真实设备，适合 CI 稳定回归。真实 AgentFI/Polymarket smoke 仍由 `scripts/real-project-suite.mjs` 承担。

### T3：代码交付闭环强化

已新增 `evaluateCodeDeliveryReadiness()`。

代码任务现在额外检查：

- 是否有可执行补丁。
- 是否有变更文件。
- 是否有影响面证据。
- 是否有验证命令。

驾驶舱新增：

```text
代码    78/100 · 待补证 · 缺 验证命令
```

### T4：记忆闭环强化

已新增 `proposePostTaskMemoryCandidate()`。

规则：

- 质量门未通过，不沉淀候选记忆，避免污染长期记忆。
- 质量门通过后，生成项目内候选记忆。
- 代码任务默认作为“代码交付模板”候选，需用户后续确认。
- 分析/网页/启动任务可形成同类任务经验。

驾驶舱可显示：

```text
候选记忆    分析报告模板：补齐 iCloser 投资报告和竞品分析
```

### T5：非技术用户向导

驾驶舱新增 `向导` 行，用普通用户能执行的语言告诉下一步。

示例：

```text
向导    回复“补齐缺口”，AI 会补：竞品分析、尽调缺口、置信度
向导    输入 d 看 diff，确认无误后输入 y 写入，再运行验证
向导    可以追问“详细点”或“提炼成报告”，AI 会复用这轮网页证据
```

### T6：任务质量报告持久化

已新增 `src/core/task-quality-report.ts`。

每个 AgentTaskLoop 任务都会尽量写入：

```text
.icloser/agent-tasks/<taskId>/quality-report.json
```

报告包含：

- 任务类型、输入、成功状态。
- Golden Path 状态。
- 质量门结果。
- 代码交付状态和就绪度。
- 证据来源、来源等级和下一步动作。
