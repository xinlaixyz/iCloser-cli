# REPL 交互升级：从日志输出器到工程驾驶舱

日期：2026-05-21  
目标：让完全不懂工程的人，也能通过 iCloser Agent Shell 完成“理解项目、生成代码、验证结果、沉淀记忆”的闭环。

## 产品判断

当前能力已经分散存在：Provider、工具执行、Evidence Store、Golden Path、Code Delivery、长期记忆。但用户正面看到的是 REPL，如果 REPL 只连续打印工具日志，用户会感觉“AI 在乱跑”，也无法判断结果是否可靠。

因此 REPL 必须成为主产品界面，而不是调试日志。

## 本轮落地

### 1. Mission Cockpit 任务驾驶舱

新增 `src/cli/mission-cockpit.ts`。

任务开始前展示：

- 用户输入是什么
- 系统判断的任务类型
- 本轮目标
- AI Provider / Model
- 是否注入记忆
- 工作目录
- 执行顺序
- 写入/执行确认点

目标是让新手不用理解“工具调用协议”，也能知道 AI 接下来要做什么。

### 2. 工具执行可解释化

`src/cli/tool-display.ts` 增加 `explainToolPurpose()`。

工具行从：

```text
read_file src/index.ts
```

升级为：

```text
read_file src/index.ts · 读取关键文件
```

用户能看懂每个工具为什么被调用，而不是只看到内部工具名。

### 3. 任务结果面板

任务结束后展示：

- 理解是否完成
- 是否取得证据
- 是否生成补丁
- 是否完成验证
- 是否使用记忆
- 证据数量、工具数量、AI 轮次、token
- 代码产物文件
- 下一步动作

这让“结果是否可信”变成可读状态，而不是让用户自己从大段文本里猜。

### 4. 继续任务不丢产物

修复 REPL 体验缺陷：代码补丁生成后，如果用户输入“继续 / 继续任务 / 下一步 / 预览 diff / 写入”，系统不再清空 `pendingFiles`，而是恢复文件写入确认面板。

这条对新手非常关键：用户不懂 `/write` 时，最自然的动作就是输入“继续”。系统必须接住这个动作，而不是丢掉刚生成的代码产物。

### 5. 长结果默认压缩

修复“AI 内容一下出来很多”的体验问题：工具任务的最终结果如果超过阈值，REPL 先展示关键前 28 行，简洁模式展示前 18 行，并提示已折叠行数。

完整内容仍进入会话历史和证据链，不影响后续追问；但用户正面看到的是可消化的摘要、任务结果面板和下一步动作。

### 6. 代码交付确认面板增强

补丁候选生成后，文件写入确认面板不再只显示文件名，而是展示：

- 变更摘要：文件数、新增/删除行数。
- 风险等级：低 / 中 / 高。
- 每个文件的可能意图。
- 建议验证命令或浏览器验收方式。

这样完全不懂代码的用户也能判断：“AI 改了什么、风险多大、点写入后该怎么验收”。

### 7. Provider 失败恢复提示

AI 调用失败不再只显示原始异常。系统会按错误类型给出恢复路径：

- 超时：建议切换更快模型，例如 `deepseek-v4-flash`，或缩小任务范围。
- 网络失败：建议运行 `ic provider test`，必要时配置代理。
- 鉴权失败：建议 `/apikey` 重新录入 Key。
- 模型不可用：建议查看模型列表并切换模型。
- 限流：建议等待后输入“继续”恢复任务。

目标是让非工程用户遇到失败时仍知道下一步，而不是停在错误栈。

### 8. 任务开始展示记忆摘要

Mission Cockpit 会在任务开始时展示本轮记忆摘要：

- 项目规则数量。
- 用户偏好数量。
- 相关历史数量。
- 待确认候选数量。
- 冲突提示。

同时这份摘要会注入本轮 AI 上下文，避免“界面说有记忆，但 AI 实际没用”的断裂。

### 9. `/help` 按用户目标分组

帮助界面从命令平铺改成产品路径：

- `setup`：配置模型和环境。
- `project`：理解和启动项目。
- `ai`：自然语言工程任务。
- `tools`：搜索、网页、命令证据。
- `code`：代码交付和验证。
- `memory`：长期记忆和项目规则。
- `collab`：提交和团队协作。
- `release`：发布前质量信任。

新手路径明确为：直接输入需求 → 看任务驾驶舱 → 看工具证据 → `/diff` → `/write` → `/verify`。

## 下一步必须继续做

1. REPL 输入后不应一次性刷屏，需要分段节奏输出：理解、取证、生成、验证、总结。
2. 代码交付成功后，自动展示 `/diff` 摘要和一键 `/write` 提示。
3. 新手模式下隐藏低价值日志，只展示“正在做什么、为什么、结果如何”。
4. 高级模式保留完整工具日志，用于工程师调试。
5. `继续` 必须恢复上一个任务的驾驶舱状态，而不是重新进入普通聊天。

## 验收标准

- 新手能在不懂命令的情况下输入自然语言任务。
- REPL 必须展示任务目标、工具目的、产物状态和下一步。
- 长结果不能刷屏，必须默认压缩展示并保留证据。
- 工具失败时必须显示失败原因和恢复建议。
- 代码任务必须在写入前展示补丁待确认状态、风险摘要和建议验证。
- 用户输入“继续”不能丢失待写入补丁，必须恢复确认面板。
- 所有新增渲染函数必须有单元测试。
